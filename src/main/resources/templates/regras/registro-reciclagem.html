Regra de Negócio

Registro de Reciclagem

Todo registro deve estar vinculado a um usuário, um ponto de coleta e um catador.

Regra obrigatória: Nenhum registro de reciclagem pode ser criado sem que os três vínculos abaixo estejam presentes e válidos no banco de dados. A ausência de qualquer um deles impede o cadastro do registro e gera erro de validação.

As três entidades obrigatórias

Usuário

Quem gerou o material reciclável e realizou a entrega no ponto de coleta.

FK id_usuario → tabela usuario
Deve estar com status ATIVO
Ponto de Coleta

Local físico onde o material foi entregue e onde a coleta ocorreu.

FK id_ponto → tabela ponto_coleta
Deve estar com status ATIVO
Catador

Profissional que recebeu o material e que poderá validar o registro.

FK id_catador → tabela usuario
Tipo deve ser CATADOR

Estrutura da tabela registro_reciclagem

CREATE TABLE registro_reciclagem (
  id_registro SERIAL PRIMARY KEY,
  id_usuario INTEGER NOT NULL, -- FK obrigatória → usuario
  id_ponto INTEGER NOT NULL, -- FK obrigatória → ponto_coleta
  id_catador INTEGER NOT NULL, -- FK obrigatória → usuario (tipo CATADOR)
  id_tipo_material INTEGER NOT NULL,
  quantidade NUMERIC(10,2) NOT NULL,
  data_registro TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  qr_code VARCHAR(255) NOT NULL UNIQUE,
  status status_reciclagem NOT NULL DEFAULT 'PENDENTE',
  ...
  CONSTRAINT fk_usuario_registro FOREIGN KEY (id_usuario) REFERENCES usuario(id_usuario),
  CONSTRAINT fk_ponto_registro FOREIGN KEY (id_ponto) REFERENCES ponto_coleta(id_ponto),
  CONSTRAINT fk_catador_registro FOREIGN KEY (id_catador) REFERENCES usuario(id_usuario)
);

As três colunas marcadas em rosa são NOT NULL com FOREIGN KEY no banco, ou seja, o próprio banco de dados rejeita qualquer INSERT que omita ou forneça um valor inexistente para qualquer uma delas.

Fluxo de criação do registro

1
Usuário inicia o registro

O usuário logado acessa "Registrar reciclagem" e informa a quantidade e o tipo de material.

2
Sistema vincula o ponto de coleta

O ponto de coleta é selecionado pelo usuário a partir dos pontos com status ATIVO disponíveis no sistema.

3
Sistema vincula o catador

O catador presente no ponto é associado ao registro no momento da criação.

4
QR Code é gerado

Um código único é gerado para o registro, permitindo que o catador faça a validação posteriormente.

5
Catador valida o registro

O catador confirma a entrega via painel. A trigger do banco credita os pontos automaticamente ao usuário.

Validações aplicadas

  • id_usuario obrigatório e existente — o usuário deve estar cadastrado e com status ATIVO.
  • id_ponto obrigatório e existente — o ponto de coleta deve estar cadastrado e com status ATIVO.
  • id_catador obrigatório e do tipo CATADOR — deve ser um usuário com tipo CATADOR vinculado ao sistema.
  • Quantidade maior que zero — a constraint CHECK quantidade > 0 é aplicada no banco.
  • Sem usuário → registro rejeitado — o banco lança erro de violação de FK.
  • Sem ponto → registro rejeitado — o banco lança erro de violação de FK.
  • Sem catador → registro rejeitado — o banco lança erro de violação de FK.
